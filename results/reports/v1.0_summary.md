## Key Findings by Article

### Article 1: State-Aware RAG

Measured on 300 synthetic queries, top_k=5, BGE-base-en-v1.5 embeddings, 3 runs.

| Technique | Recall@5 | MRR | Latency (mean) |
|-----------|---------|-----|----------------|
| Naive RAG (baseline) | 0.388 | 0.417 | 705 ms |
| HyDE | 0.503 (+11.4 pp) | 0.481 | 1060 ms (+355 ms) |
| Query Decomposition | 0.474 (+8.6 pp) | 0.421 | 1516 ms (+811 ms) |
| HyDE + Decomposition | 0.509 (+12.0 pp) | 0.437 | 2611 ms (+1906 ms) |

Key finding: HyDE alone earns most of the available recall lift; layering decomposition
on top adds ~0.6 pp recall at 2.5× the latency. Graph RAG is implemented in code but
not in the canonical benchmark runner — qualitative finding (traversal-path strings
inflate answers and hurt faithfulness) holds.
Source: `results/data/article_01_benchmarks.json`

### Article 2: Advanced Retrieval

Measured on 30 hand-graded queries, 211 chunks, top_k=5, BGE-base-en-v1.5 embeddings, 3 runs.

| Configuration | Recall@5 | MRR | Latency (mean) |
|---|---|---|---|
| Dense-only (baseline) | 0.761 | 0.739 | 866 ms |
| Hybrid (BM25 + dense + RRF) | 0.761 | 0.648 | 871 ms |
| Hybrid + FlashRank reranking (top-20 → top-5) | 0.683 | 0.769 | 995 ms |

Key finding: on this corpus hybrid did not lift recall over dense-only — BGE-base-en-v1.5
already covers the keyword/paraphrase split, so BM25 found the same docs and shuffled
ranks (MRR drops 0.739 → 0.648). The cross-encoder reranker recovered ranking quality
(MRR → 0.769) but introduced cross-encoder mispredictions that dropped Recall@5 to 0.683.
Hybrid+rerank is a recall/ranking trade-off on small high-quality corpora, not a stack
of wins. Metadata pre-filter and semantic chunking remain implemented but were dropped
from the canonical runner (commit 279e41f) to keep the apples-to-apples comparison clean.
Source: `results/data/article_02_benchmarks.json`

### Article 6: LLM Ops

Measured on 100 queries; cache and routing benchmarks isolate different baselines.

| Dimension | Baseline | With optimisation | Reduction |
|---|---|---|---|
| Routing (Groq-8B for simple, GPT-4o for complex) | $0.1935 (GPT-4o for everything) | $0.00228 | 98.8% |

Routing distribution: 97% of queries classified simple → Groq-8B, 3% complex → GPT-4o.
Cache hit p50 latency 0.51 ms; miss p50 latency 415 ms.

### Article 9: Deep Learning

Measured on M4 Pro hardware. Fine-tune used (query, answer-summary) pairs — wrong objective for retrieval; result is a deliberate negative finding kept in the public dataset.

| Technique | Metric | Result |
|-----------|--------|--------|
| BGE fine-tuning (wrong objective, regression) | Recall@5 | 0.729 → 0.622 (−10.7 pp) |
| BGE fine-tuning | Recall@1 | 0.463 → 0.396 (−6.8 pp) |
| BGE fine-tuning | Recall@3 | 0.655 → 0.571 (−8.5 pp) |
| torch.compile on MPS | Embed latency | 23.97 ms → 26.12 ms (0.92×, no win on Apple Silicon) |
| INT8 dynamic quantisation | Embed latency | 23.49 ms → 48.95 ms (0.48×, ~2.08× SLOWER on M4) |
| INT8 model size | | 437.9 MB → 109.5 MB (4.0× smaller) |
| INT8 accuracy impact | Cosine similarity | 0.983 |

Why INT8 hurts latency on M4: PyTorch's QNNPACK INT8 kernels are tuned for x86 VNNI;
Apple Silicon takes the size win and pays a latency tax. Why fine-tuning regressed:
training pairs (query, answer-summary) did not mirror the inference task (query, doc-chunk).
Both findings are kept in the canonical dataset to document the failure mode.
Source: `results/data/article_09_benchmarks.json`, `pytorch_optimizations.json`

## Limitations and Future Work

1. **Red-team coverage**: 18.3% overall block rate leaves sophisticated adversarial
   prompts (base64-encoded, ROT13, unicode homoglyphs) partially unaddressed.
   Production systems should add LLM-based semantic classification for all inputs.

2. **Fine-tuning objective**: The (query, answer-summary) pair format used for the
   v1.0 BGE fine-tune did not mirror the inference task (query, doc-chunk) and
   produced a Recall@5 regression (0.729 → 0.622). A future revision should retrain
   on (query, doc-chunk) pairs from the same chunking pipeline used at retrieval time.

3. **K8s benchmarks simulated**: The scaling results in Article 8 use simulated
   data (Locust mock mode). Real cluster benchmarks require cloud infrastructure.

4. **GDPR Art.17 gap**: Right to erasure is not implemented. The audit logger
   stores SHA-256 hashed inputs (not plaintext) but does not support deletion.

5. **Graph RAG faithfulness**: The NetworkX graph RAG implementation generates
   traversal path strings that inflate answers. A production implementation would
   filter traversal metadata from the context window.

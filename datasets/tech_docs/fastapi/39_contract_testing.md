# Contract Testing with FastAPI

Contract testing is a critical practice in microservices architecture that ensures services can communicate reliably without requiring all components to be available simultaneously. By defining and validating explicit contracts between consumers and providers, teams can prevent integration errors, reduce testing overhead, and accelerate deployment pipelines. In the context of FastAPI, contract testing leverages tools like **Pact** for consumer-driven contracts and **OpenAPI** for schema validation. This documentation explores how to implement contract testing in production-grade FastAPI applications, with code examples, practical use cases, and comparisons to alternative approaches.

---

## Consumer-Driven Contracts: The Foundation of Reliable Integration

Consumer-driven contracts (CDCs) define the expectations of a service consumer (e.g., a frontend or microservice) about the behavior of a provider (e.g., a backend API). This approach ensures the provider meets the consumer’s requirements and reduces the risk of breaking changes during integration. In FastAPI, CDCs can be implemented using **Pact**, a framework that records interactions between services and verifies they remain compatible over time.

### Why Use Consumer-Driven Contracts?

- **Decouples Development**: Teams can develop services independently while ensuring compatibility.
- **Faster Feedback**: Tests catch contract violations early in the CI/CD pipeline.
- **Production Parity**: Simulates real-world interactions with mock providers or consumers.

### Example: Pact Consumer Test in Python

Below is a Pact test for a FastAPI consumer that expects a `/users/{id}` endpoint to return a user profile:

```python
# tests/consumer_pact_test.py
import pytest
from pact import Consumer, HasPact, Provider

pact = Consumer('UserConsumer').has_pact_with(Provider('UserProvider'), host_name='localhost', port=8080)

@pact.given('a user with ID 123 exists')
@pact.upon_receiving('a request for user 123')
@pact.will_return(status=200, body={'id': 123, 'name': 'Alice'})
def test_user_provider():
    from requests import get
    response = get('http://localhost:8080/users/123')
    assert response.json()['id'] == 123
    assert response.json()['name'] == 'Alice'
```

This test defines a **contract** between the consumer and provider. When executed, Pact generates a mock provider server and verifies the consumer behaves as expected. The output is a **Pact file** that the provider can later verify.

## OpenAPI Contract Validation: Ensuring Schema Compliance

FastAPI natively supports **OpenAPI** (Swagger) for API documentation and schema validation. By treating the OpenAPI schema as a contract, teams can enforce strict input/output validation and automate testing.

### Validating Requests and Responses Against OpenAPI

FastAPI automatically validates requests and responses based on the schema defined in your code. For example:

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    id: int
    name: str
    email: str | None = None

@app.get("/users/{user_id}", response_model=User)
def get_user(user_id: int):
    return {"id": user_id, "name": "Alice", "email": "alice@example.com"}
```

Here, FastAPI ensures:
- The response conforms to the `User` model.
- Any invalid data (e.g., a string instead of an integer for `id`) triggers a 422 error.

### Automating Schema Validation with Tools

Tools like `openapi-core` or `Swagger UI` can validate external requests against your OpenAPI schema. For example:

```bash
# Generate OpenAPI schema
curl http://localhost:8000/openapi.json > openapi.json

# Validate with openapi-core
python -m openapi_core validate --spec openapi.json --validator swagger --data request.json
```

This approach is ideal for **provider-side validation**, ensuring the API adheres to its documented contract.

## Pact Implementation in FastAPI: Provider Verification

Once a Pact file is generated by the consumer, the provider must verify it. In FastAPI, this can be done using the FastAPI test client and Pact’s verification framework.

### Example: Provider Test with Pact

```python
# tests/provider_pact_verification.py
import pytest
from fastapi.testclient import TestClient
from pact import Verifier

from main import app  # Your FastAPI app

client = TestClient(app)

def test_pact_verification():
    verifier = Verifier(provider='UserProvider', consumer='UserConsumer')
    # Pact file path from consumer test output
    success, _ = verifier.verify_pact(
        'http://localhost:8080',
        'consumer_pact.json',
        provider_base_url='http://localhost:8000',
        test_client=client
    )
    assert success
```

This test loads the Pact file and runs all interactions against the FastAPI provider. If the provider fails to meet the contract (e.g., returns an incorrect status code or body), the test will fail, signaling a breaking change.

## Best Practices for Production-Grade Contract Testing

### 1. **Version Contracts Explicitly**
- Include version numbers in contract filenames (e.g., `user-api-v1-pact.json`).
- Use semantic versioning to manage breaking changes.

### 2. **Integrate with CI/CD Pipelines**
- Run consumer and provider tests in parallel to reduce feedback time.
- Example GitHub Actions workflow:

```yaml
jobs:
  consumer:
    runs-on: ubuntu-latest
    steps:
      - name: Run Pact Consumer Tests
        run: pytest tests/consumer_pact_test.py
  provider:
    runs-on: ubuntu-latest
    steps:
      - name: Verify Pact
        run: pytest tests/provider_pact_verification.py
```

### 3. **Use Pact Broker for Contract Management**
- Store and share contracts using [Pact Broker](https://docs.pact.io/pact_broker/).
- Automate contract publishing and verification with webhooks.

### 4. **Handle Dynamic Data with Pact Matchers**
- Use matchers to avoid brittle tests for dynamic values (e.g., timestamps):

```python
from pact import Term, EachLike

@pact.will_return(
    status=200,
    body={
        'id': 123,
        'created_at': Term(pattern=r'\d{4}-\d{2}-\d{2}', generate='2023-01-01'),
        'roles': EachLike(item={'id': Term(r'\d+', '1')})
    }
)
```

## Troubleshooting Common Issues

### 1. **Pact File Mismatches**
- **Symptoms**: Provider fails to verify a contract.
- **Solution**: Ensure the consumer and provider use the same contract version. Use Pact Broker to manage versions.

### 2. **OpenAPI Schema Discrepancies**
- **Symptoms**: Valid requests are rejected by FastAPI.
- **Solution**: Double-check the `response_model` and `request_model` definitions. Use `pydantic`’s `Config` to relax schema rules if needed.

### 3. **Missing Optional Fields in Responses**
- **Symptoms**: Consumer expects a field that is occasionally absent.
- **Solution**: Define optional fields in the OpenAPI schema using `Union[None, Type]` or `Field(default=None)`.

## Comparisons: Pact vs. OpenAPI Validation

| **Feature**               | **Pact**                          | **OpenAPI Validation**          |
|---------------------------|-----------------------------------|---------------------------------|
| **Scope**                 | Interaction-level testing         | Schema-level validation         |
| **Tooling**               | Pact CLI, Pact Broker               | Swagger UI, openapi-core      |
| **Best For**              | Microservices with CDCs           | Documentation and provider-side validation |
| **Integration Difficulty**| Requires consumer/provider tests  | Built into FastAPI              |

Choose **Pact** for complex, consumer-driven workflows and **OpenAPI** for lightweight schema validation.

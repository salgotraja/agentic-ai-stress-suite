# Spring GraphQL

GraphQL is a modern query language and runtime for APIs that allows clients to request exactly the data they need, making it a powerful alternative to traditional REST APIs. Spring GraphQL, part of the broader Spring ecosystem, integrates GraphQL capabilities into Spring-based applications, enabling developers to build efficient, flexible, and scalable APIs.

This guide covers the key concepts, implementation patterns, and best practices for working with Spring GraphQL, focusing on schema design, resolvers, queries, mutations, subscriptions, and endpoint configuration.

---

## Key Concepts

### GraphQL Schema

The foundation of any GraphQL service is the **GraphQL schema**, which defines the types, queries, mutations, and subscriptions available in the API. Schema-first development is common in Spring GraphQL, where the schema is defined using the `.graphqls` file format and is then used to generate Java interfaces and classes automatically.

Schema definition example (`schema.graphqls`):

```graphql
type Query {
    user(id: ID!): User
    users: [User]
}

type Mutation {
    createUser(input: UserInput!): User
}

type Subscription {
    userCreated: User
}

type User {
    id: ID!
    name: String!
    email: String!
}

input UserInput {
    name: String!
    email: String!
}
```

### Resolvers

Resolvers are Java methods that fetch data for the GraphQL operations. In Spring GraphQL, resolvers are typically implemented as Spring components that implement the types generated by the schema. For example, the `Query` type defined above can be implemented as follows:

```java
@Component
public class QueryResolver implements Query {

    @Override
    public User user(String id) {
        return User.findById(id);
    }

    @Override
    public List<User> users() {
        return UserRepository.findAll();
    }
}
```

Resolvers can also be annotated with `@SchemaMapping` for more flexibility and fine-grained control over schema-to-code mapping.

## GraphQL Endpoint and Bootstrapping

Spring GraphQL integrates with Spring Boot, making it easy to expose a GraphQL endpoint. By default, the GraphQL endpoint is available at `/graphql`. You can customize this endpoint and enable GraphiQL (an in-browser IDE for interacting with GraphQL APIs) by configuring the `application.yml` file:

```yaml
spring:
  graphql:
    graphiql:
      enabled: true
    endpoint:
      mapping: /api/graphql
```

To enable GraphQL support in your Spring Boot application, add the following dependency to your `pom.xml` or `build.gradle`:

Maven:
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-graphql</artifactId>
</dependency>
```

Gradle:
```groovy
implementation 'org.springframework.boot:spring-boot-starter-graphql'
```

Once the dependency is in place, the Spring GraphQL engine auto-detects schema files and generates the necessary code.

## Queries and Mutations

Queries are used to fetch data, and mutations are used to modify data. Both are declared in the schema and implemented via Java classes. Let's explore how to implement mutations in practice.

Mutation resolver example:

```java
@Component
public class MutationResolver implements Mutation {

    @Override
    public User createUser(UserInput input) {
        return UserRepository.save(input);
    }
}
```

GraphQL query example using GraphiQL or a client like Postman:

```graphql
query {
  user(id: "123") {
    id
    name
    email
  }
}
```

GraphQL mutation example:

```graphql
mutation {
  createUser(input: { name: "Alice", email: "alice@example.com" }) {
    id
    name
    email
  }
}
```

**Best Practice**: Always validate input in mutations. Use `@InputValue` or `@Argument` annotations to extract input values and validate them before performing business operations.

## Subscriptions

Subscriptions allow clients to subscribe to real-time data updates. Spring GraphQL supports WebSockets for subscriptions via the Apollo library or custom WebSocket configurations.

Example schema for subscriptions:

```graphql
type Subscription {
    userCreated: User
}
```

Subscription resolver implementation:

```java
@Component
public class SubscriptionResolver implements Subscription {

    @Override
    public Flux<User> userCreated() {
        return UserRepository.userCreated().map(User::new);
    }
}
```

GraphQL subscription example:

```graphql
subscription {
  userCreated {
    id
    name
    email
  }
}
```

**Tip**: Subscriptions are ideal for real-time features such as chat, notifications, and live updates. Ensure that your data source supports event-driven mechanisms like Kafka or Redis for efficient updates.

## Schema Generation and Mapping

Spring GraphQL provides schema-first and code-first design options. In schema-first design, the schema is the source of truth, and Spring generates Java interfaces based on it. This approach is preferred in larger teams and projects with strict API contracts.

For schema-first, place `.graphqls` files in the `src/main/resources/graphql` directory. Spring GraphQL processes these files and generates `@SchemaMapping` annotated interfaces for each type.

Example schema:

```graphql
type Book {
    id: ID!
    title: String!
    author: String!
}
```

Generated interface:

```java
public interface Book {
    String getId();
    String getTitle();
    String getAuthor();
}
```

You can then implement this interface in a Spring component:

```java
@Component
@SchemaMapping(typeName = "Query", field = "books")
public class BookResolver implements Book {

    @Override
    public List<Book> books() {
        return BookService.findAll();
    }
}
```

This pattern ensures that your Java code aligns closely with the schema, reducing the possibility of mismatched types and fields.

## Best Practices

### 1. Schema Design

- **Use descriptive and consistent naming** for types and fields.
- **Avoid overloading** a single resolver with multiple responsibilities.
- **Use input types** for mutations and complex queries to keep the schema clean.

### 2. Error Handling

GraphQL errors should be returned in a structured format. Spring GraphQL supports returning exceptions as `GraphQLError` objects. For example:

```java
@Override
public User user(String id) {
    return UserRepository.findById(id)
            .orElseThrow(() -> new GraphQLError("User not found with ID: " + id));
}
```

Always include error messages that are meaningful to the client and avoid exposing internal implementation details.

### 3. Performance Optimization

- **Implement batching and caching** for data fetching to avoid N+1 queries.
- **Use DataLoader** to batch and cache requests to external services or databases.
- **Leverage schema directives** for custom behaviors like rate limiting or access control.

### 4. Data Fetching

For efficient data fetching, prefer the following:

- **Use reactive types (`Mono`, `Flux`) for non-blocking I/O**.
- **Avoid blocking operations** in resolvers, especially in production.
- **Structure your data layer** to support GraphQL’s hierarchical nature.

## Comparison with REST APIs

While REST APIs are resource-oriented and rely on multiple endpoints, GraphQL offers a single endpoint with flexible queries. Here's a comparison:

| Feature               | REST API                        | GraphQL                         |
|-----------------------|----------------------------------|----------------------------------|
| Data Fetching         | Multiple endpoints              | Single endpoint with queries     |
| Over-fetching         | Likely                          | Avoidable                        |
| Under-fetching        | Likely                          | Avoidable                        |
| Versioning            | Versioned endpoints             | Schema evolution, no versioning  |
| Real-time Updates     | Polling or WebSockets           | Subscriptions                    |
| Client Flexibility     | Fixed payloads                 | Customizable payloads            |

GraphQL is ideal for complex, data-intensive applications, while REST remains a good choice for simple APIs or when strict versioning is required.

## Troubleshooting and Common Pitfalls

### 1. Schema Validation Errors

- **Problem**: Schema does not match resolver methods.
- **Solution**: Use `@SchemaMapping` annotations or ensure the generated interfaces match the schema.

### 2. Missing Resolvers

- **Problem**: Schema types are not implemented.
- **Solution**: Ensure that all schema types have corresponding resolver components.

### 3. Performance Issues

- **Problem**: Slow queries due to N+1 patterns.
- **Solution**: Implement batching and caching using `DataLoader`.

### 4. Subscription Not Working

- **Problem**: Clients cannot connect to the subscription endpoint.
- **Solution**: Enable WebSocket support and configure CORS properly.

## Real-World Use Case

**Example**: E-commerce Platform with Product Catalog and User Notifications

- **GraphQL schema** includes `Product`, `User`, and `Order` types.
- **Queries** allow fetching products and orders.
- **Mutations** support placing orders and updating user details.
- **Subscriptions** notify users in real-time when new orders arrive.

Schema snippet:

```graphql
type Product {
    id: ID!
    name: String!
    price: Float!
}

type Query {
    product(id: ID!): Product
    products: [Product]
}

type Mutation {
    createProduct(input: ProductInput!): Product
}
```

Resolver component:

```java
@Component
public class ProductQueryResolver implements Query {

    private final ProductService productService;

    public ProductQueryResolver(ProductService productService) {
        this.productService = productService;
    }

    @Override
    public Product product(String id) {
        return productService.getProduct(id);
    }

    @Override
    public List<Product> products() {
        return productService.getAllProducts();
    }
}
```

Spring GraphQL enables developers to build modern, efficient, and flexible APIs using the power of GraphQL. By leveraging schema-first design, resolvers, and reactive programming, Spring makes it easy to integrate GraphQL into enterprise applications.

By following the patterns and examples outlined in this guide, you can build scalable and maintainable GraphQL APIs with Spring that meet the needs of your users and business.

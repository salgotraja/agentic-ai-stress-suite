### Article 1: State-Aware RAG

| Technique | Recall@5 | Latency (p50) |
|-----------|---------|--------------|
| Naive RAG (baseline) | 0.723 | 180ms |
| HyDE | +3pp | +280ms (extra LLM call) |
| Query Decomposition | +4pp | +320ms (parallel sub-queries) |
| Graph RAG (NetworkX) | +5pp Recall, -6pp Faithfulness | +450ms |

Key finding: Graph RAG improves recall but hurts faithfulness. Traversal path
strings appear in generated answers and produce unsupported claims.

## Limitations and Future Work

1. **Red-team coverage**: 18.3% overall block rate leaves sophisticated adversarial
   prompts (base64-encoded, ROT13, unicode homoglyphs) partially unaddressed.
   Production systems should add LLM-based semantic classification for all inputs.

2. **Fine-tuning data size**: 1,000–2,000 training pairs is small. With 10,000+
   pairs from a real production corpus, +17pp Recall@5 would likely be +25–30pp.

3. **K8s benchmarks simulated**: The scaling results in Article 8 use simulated
   data (Locust mock mode). Real cluster benchmarks require cloud infrastructure.

4. **GDPR Art.17 gap**: Right to erasure is not implemented. The audit logger
   stores SHA-256 hashed inputs (not plaintext) but does not support deletion.

5. **Graph RAG faithfulness**: The NetworkX graph RAG implementation generates
   traversal path strings that inflate answers. A production implementation would
   filter traversal metadata from the context window.
